推荐赛道：**课程智能体**

## 0. 文档边界与核心判断

本文档按“可开发、可演示、可答辩、可扩展”的完整项目规格编写，不按个人开发者身份压缩项目完整性。项目目标不是做一个医学问答机器人，而是做一个**诊断学课程训练智能体**：学生通过多轮问诊、查体申请、辅助检查申请、诊断提交和复盘反馈完成一次 OSCE 式临床思维训练。

本项目的数据策略默认不依赖校内教师或私有医院数据，而是使用**公开数据、公开病例、公开开源项目和自行结构化加工病例**。但必须说明：没有医学教师参与时，系统只能定位为**教学原型 / 比赛作品 / 模拟训练工具**，不能声称已完成正式课程医学审核。

赛事公开通知强调“人工智能+医学教学”融合创新、医学教育数字化转型和智能化教学能力提升，这决定项目必须体现教学载体、教学环节、实施路径和可展示效果，而不是单纯展示 Agent 技术。赛事通知地址：`https://www.cfxy.cn/yxb/tzgg/81c3510906174816a858d450261d12fb.htm`。 ([cfxy][1])

# 1. 项目概述

## 1.1 项目定位

**问诊推理舱**是一个面向医学生的诊断学临床思维训练智能体。系统通过虚拟标准化病人模拟、病例状态控制、问诊意图识别、查体/检查工具调用、OSCE rubric 评分和个性化反馈，帮助学生练习“如何一步步问出关键病史并形成诊断推理”。

项目不解决真实患者诊疗问题，不提供真实医疗建议，不替代医生和教师。所有病例均为公开数据改写或虚构教学病例，系统输出仅用于医学教育训练。

## 1.2 一句话价值主张

**让医学生不只是“知道疾病答案”，而是在智能体陪练中完成一次可记录、可评分、可复盘、可迭代的诊断学问诊训练。**

> 仓库当前状态 (2026-04-23)：已完成项目骨架（FastAPI `/health` + LangGraph 依赖）、5 条公开数据源登记、约 1.5 GB 原始公开数据下载；`data/cases` / `data/rubrics` / `data/processed` 尚未产出结构化病例，是 Step 2–Step 3 的下一阶段交付物。

## 1.3 推荐赛道

**课程智能体。**

不要报“临床技能训练智能体”。原因是临床技能训练赛道天然要求高仿真临床场景，容易被评委追问医学准确性、临床责任、操作技能真实性。课程智能体更适合本项目，因为它可以绑定“诊断学 / 临床思维训练 / 内科学见习前训练”课程，重点展示教学目标、课程内容、评价方式和学生学习闭环。

## 1.4 核心用户

| 用户          | 角色定位      | 主要需求                           |
| ----------- | --------- | ------------------------------ |
| 医学生         | 训练者       | 练习问诊、查体选择、辅助检查选择、鉴别诊断和诊断表达     |
| 项目开发者 / 管理者 | 系统维护者     | 管理公开病例、配置 rubric、查看训练日志、评估系统表现 |
| 未来课程教师      | 审核者 / 使用者 | 审核病例、调整评分表、查看班级共性问题            |
| 评委          | 体验者       | 判断项目是否真实解决医学教学问题、是否有智能体特征和可推广性 |

# 3. 总体产品设计

## 3.1 核心训练流程

学生进入系统后，完成如下流程：

```text
选择课程模块
→ 进入病例
→ 阅读主诉
→ 多轮问诊
→ 请求查体
→ 请求辅助检查
→ 提交主要诊断
→ 提交鉴别诊断
→ 提交诊断依据
→ 系统评分
→ 系统反馈
→ 生成学习报告
→ 进入下一病例推荐
```

系统在学生提交诊断前不能直接泄露标准答案。系统只能根据学生问到的内容逐步开放病例信息。

## 3.2 产品核心差异

| 普通医学问答系统    | 问诊推理舱                   |
| ----------- | ----------------------- |
| 回答医学知识问题    | 模拟完整问诊训练                |
| 学生问什么，系统答什么 | 系统维护病例状态和训练阶段           |
| 不记录过程质量     | 记录问诊、查体、检查和诊断推理路径       |
| 主要评价最终答案    | 评价全过程：问诊、检查选择、鉴别诊断、推理表达 |
| 知识库是核心      | OSCE 训练状态机是核心，RAG 只是辅助  |
| 无进化机制       | 从训练日志中沉淀教学 Skill        |

## 3.3 项目边界

### 必须做

| 功能       | 说明                             |
| -------- | ------------------------------ |
| 虚拟标准化病人  | 能根据结构化病例多轮回答学生问题               |
| 问诊意图识别   | 判断学生是在问病史、请求查体、请求检查、请求提示还是提交诊断 |
| 信息披露控制   | 只回答学生问到的信息，不提前泄露诊断             |
| 查体/检查工具  | 学生请求后返回结构化查体或辅助检查结果            |
| OSCE 评分  | 按病例 rubric 对问诊、检查、诊断、推理评分      |
| 个性化反馈    | 输出优势、漏项、错误推理、下一步训练             |
| 公开数据来源追踪 | 每个病例和知识点都要有来源记录                |
| 训练日志     | 保存学生路径、系统回复、工具调用和评分结果          |
| 受控进化     | 从日志中生成教学 Skill 候选，经审核后启用       |

### 不做

| 不做项           | 原因                    |
| ------------- | --------------------- |
| 真实患者导诊        | 医疗风险高，偏离课程智能体         |
| 用药建议          | 合规风险高                 |
| 接入医院 HIS/EMR  | 个人开发无法获得稳定授权          |
| 真实临床决策支持      | 责任边界过重                |
| 影像识别/手术模拟     | 工程成本高，且不利于稳定演示        |
| 任意代码执行型 Agent | 医学教学系统不需要，安全风险过高      |
| 自动修改医学事实      | 自我进化只能改教学策略，不能改病例标准答案 |

| `reasoning_points`       | 关键诊断依据         |
| `rubric`                 | 评分表            |
| `safety_notes`           | 教学模拟声明和真实诊疗限制  |
| `source_attribution`     | 来源地址、许可、加工说明   |

### 4.1.1 病例顶层 Pydantic 模型（v1.1 新增）

上表是人读概览，下面给出 v1.1 机读 schema。所有病例 JSON 必须能通过此模型的 `model_validate` 才能进入 `data/cases/`。

```python
from __future__ import annotations
from typing import Literal
from pydantic import BaseModel, Field

SchemaVersion = Literal["1.1"]

CourseModule = Literal[
```

**设计要点**

- `case_id` 强制 `^[a-z0-9_]+_\d{3}$`（如 `appendicitis_001`、`hyperthyroid_001`），便于跨表拼接稳定外键。
- `course_module` 使用 `Literal` 枚举而非自由字符串，对齐 §4.1 概览表的"课程模块"维度，保证 §9 RAG 检索按模块过滤时不会因写法差异漏召回。
- `rubric_ref` 指向独立 `rubric_id`（而非内嵌 rubric 对象），允许同一病例同时挂"初学者版 / 高年级版"两套 rubric。
- `source_attribution` 与已存在的 [sources.json](clinical-osce-agent/data/attribution/source_registry/sources.json) 通过 `source_id` 关联，避免来源字段在病例与登记表中双写。
- `schema_version` 固定 `Literal["1.1"]`，Step 2 落地后若需向 v1.2 演进，可通过 Discriminated Union 做 schema 兼容层。

### 4.1.3 HistoryTaking 与 HiddenFact（信息披露粒度）

v1.0 §4.1 的 `hidden_history` 仅一行描述。v1.1 把它拆成**条目化披露表**：每条 `HiddenFact` 绑定 topic、slot、触发意图、披露规则和 rubric 关联，是整个"信息披露控制 + 规则评分"的共同数据底座。

```python
HistoryTopic = Literal[
    "现病史", "既往史", "家族史", "月经史",
    "生殖史", "个人史", "用药史", "过敏史",
    "社会史", "系统回顾", "ICE",
]

# 对齐 MediTOD dialog_state.positive_symptom 及诊断学问诊六元素
HistorySlot = Literal[
    "onset", "duration", "location", "character",
    "radiation", "severity", "aggravating", "relieving",
    "progression", "associated_symptom", "frequency",
    "timing", "context", "negation",
]

BlockingRule = Literal[
    "reveal_on_direct_question",   # 学生直接问到才披露 (默认)
    "reveal_after_stage",          # 进入指定阶段后即使没问也会主动给出 (仅用于 ICE)
    "never_auto_reveal",           # 非直接问到绝对不披露, 且安全 prompt 额外检查
]

class HiddenFact(BaseModel):
    fact_id: str                              # 形如 "appendicitis_001.hf_03"
    topic: HistoryTopic
    slot: HistorySlot | None = None           # 仅 topic="现病史"/"系统回顾" 时必填
    canonical_answer: str                     # 标准答案原文
    variants: list[str] = []                  # 同义表达, 供 patient_response_node 选用
    trigger_intents: list[str]                # 命中该 fact 的 intent 标签 (≥1)
    blocking_rule: BlockingRule = "reveal_on_direct_question"
    linked_rubric_items: list[str] = []       # 关联 RubricItem.item_id (≥0)

class HistoryTaking(BaseModel):
    present_illness_summary: str              # 患者第一句话可参考但不得全部读出
    hidden_facts: list[HiddenFact]
    past_medical_history: str | None = None
    surgery_injury_history: str | None = None
    transfusion_history: str | None = None
    infection_history: str | None = None
    allergy_history: str | None = None
    personal_history: str | None = None
    menstrual_history: str | None = None
    reproductive_history: str | None = None
    family_history: str | None = None
```

**设计要点**

- `hidden_facts[]` 是病例的**原子事实单元**。§6 `patient_response_node` 必须仅从此派生回答，LLM 不得编造；任何新披露都必须在 `revealed_facts` 状态中记录 `fact_id`。
- `trigger_intents` 是 §4.5.2 `MatchRule.kind = "intent_keyword"` 评分规则的直接输入。
- `blocking_rule = "reveal_after_stage"` 仅用于 ICE：许多 OSCE 评分要求医生主动询问患者想法/担忧/期望，若学生始终不问，可在问诊末期由系统主动暴露，便于评分。
- `never_auto_reveal` 适用于"关键诊断线索"（如阑尾炎中的"转移性右下腹痛"），Prompt 层额外加防护：即便学生反问"还有什么没告诉我的吗"也不得主动补充。
- 传统病史段（`past_medical_history` 等）保留为顶层字符串：学生往往会一次性问"既往有什么病？"，不必拆成 slot 粒度。

### 4.1.4 PhysicalExamBundle / AuxiliaryTestBundle

对齐 [EasyMED evaluation.py case_template](clinical-osce-agent/references/EasyMED/EasyMED/evaluation.py#L54-L77) 的 `mustPhysicalExams / optionalPhysicalExams / physicalExams` 三段式，但用更严格的 Pydantic 类型约束，并增加 `forbidden_items` 承接 v1.0 §4.3 的"不合理检查提示"。

```python
class PhysicalExamItem(BaseModel):
    exam_code: str = Field(..., pattern=r"^[a-z][a-z0-9_]*(\.[a-z][a-z0-9_]*)+$")
    exam_name_cn: str                         # 腹部反跳痛 / 墨菲征 / Blumberg 征
    result: str                               # 标准结果描述
    is_abnormal: bool                         # 评分时用来判定异常发现是否被追问
    linked_rubric_items: list[str] = []

class PhysicalExamBundle(BaseModel):
    must_items: list[PhysicalExamItem]        # 教学上必须请求的查体
    optional_items: list[PhysicalExamItem]    # 合理但非必须
    # 学生请求但本病例未提供的, 由 physical_exam_node 统一返回 "未提供"

TestCategory = Literal["实验室", "影像", "心电", "内镜", "病理", "其他"]

class AuxiliaryTestItem(BaseModel):
    test_code: str = Field(..., pattern=r"^(lab|img|ecg|endo|path|other)\.[a-z][a-z0-9_]*$")
    test_name_cn: str
    category: TestCategory
    invasiveness: Literal["无创", "微创", "有创"]
    cost_hint: Literal["基础", "中等", "昂贵"]
    result: str
    is_abnormal: bool
    linked_rubric_items: list[str] = []

class AuxiliaryTestBundle(BaseModel):
    must_items: list[AuxiliaryTestItem]
    optional_items: list[AuxiliaryTestItem]
    forbidden_items: list[str] = []           # test_code 列表, 命中扣分
```

**设计要点**

- `exam_code` / `test_code` 是**稳定字符串 ID**（形如 `abd.palpation.rebound`、`lab.cbc`、`img.abd_us`），用于 §13 `requested_exams / requested_tests` 外键，以及 §15 回归测试的精确断言。
- `forbidden_items` 存的是 `test_code`（字符串），避免跟 `must_items / optional_items` 形成结构循环引用。
- `is_abnormal` 用于 §4.5.2 `MatchRule.kind = "reasoning_coverage"` 判定"异常发现是否被推理引用"。

### 4.1.5 DiagnosisSpec 与 ReasoningPoint

```python
ReasoningKind = Literal["支持", "排除", "鉴别"]

class ReasoningPoint(BaseModel):
    point_id: str                              # 形如 "appendicitis_001.rp_02"
    statement: str                             # "右下腹固定压痛 + 白细胞升高 支持阑尾炎"
    kind: ReasoningKind
    required_evidence: list[str] = []          # fact_id / exam_code / test_code 的混合列表
    weight: int = Field(..., ge=1, le=10)

class DifferentialDiagnosis(BaseModel):
    disease_name: str
    icd10_hint: str | None = None
    expected_action: Literal["支持", "排除"]
    key_distinction: str                       # "与阑尾炎区别: 无转移性疼痛"

class DiagnosisSpec(BaseModel):
    main_diagnosis: str
    main_diagnosis_synonyms: list[str] = []    # "急性阑尾炎" / "Acute appendicitis" / "阑尾炎"
    icd10_hint: str | None = None
    differential_diagnoses: list[DifferentialDiagnosis] = Field(..., min_length=2)
    reasoning_points: list[ReasoningPoint]
    suggested_next_steps: str                  # 教学模拟下的下一步方向 (非真实用药)
```

**设计要点**

- `main_diagnosis_synonyms` 让学生写口语、英文、缩写也能命中（通过 §4.5.2 `MatchRule.kind = "diagnosis_concept"` 做模糊匹配）。
- `reasoning_points[].required_evidence` 与 §4.5.2 `reasoning_coverage` 规则直接对接：学生的推理表达覆盖多少比例的 evidence 决定该项得分。
- `suggested_next_steps` 禁止出现具体药名和剂量（与 §14 安全边界呼应）。

## 4.2 OSCE 训练模块

### 目标

让学生像面对标准化病人一样进行问诊，而不是直接索取答案。

### 功能

| 功能     | 描述                  | 优先级 |
| ------ | ------------------- | --- |
| 进入病例   | 展示主诉和初始任务           | P0  |
| 多轮问诊   | 学生自由输入问题，虚拟患者回答     | P0  |
| 当前阶段显示 | 显示问诊、查体、检查、诊断、反馈等阶段 | P0  |
| 已收集线索  | 自动列出学生已获得的关键信息      | P0  |
| 诊断假设草稿 | 学生可记录自己的初步假设        | P1  |
| 苏格拉底提示 | 学生卡住时给过程提示，不直接给答案   | P1  |
| 训练计时   | 模拟 OSCE 限时环境        | P2  |

### 回答原则

虚拟患者必须遵守：

1. 只扮演患者，不扮演医生；
2. 只回答病例中存在的信息；
3. 不主动说出未被问到的关键事实；
4. 不在诊断前透露标准诊断；
5. 对非本病例问题保持患者视角；
6. 对真实医疗问题触发安全边界；
7. 对学生要求“直接告诉我答案”时，只能提示继续问诊或提交诊断。

## 4.3 查体与辅助检查模块

### 目标

把“学生问一句，系统答一句”升级为“学生主动选择检查，系统按病例返回结果”。

### 功能

| 功能       | 描述              | 优先级 |
| -------- | --------------- | --- |
| 查体申请     | 学生选择或输入查体项目     | P0  |
| 检查申请     | 学生选择血常规、尿常规、影像等 | P0  |
| 结果返回     | 只返回病例库中定义的结果    | P0  |
| 不合理检查提示  | 过度或无关检查可提示其教学意义 | P1  |
| 成本/侵入性标签 | 标记检查是否基础、昂贵、有创  | P2  |

### 检查返回原则

* 结构化字段优先，LLM 只负责表达；
* 不允许 LLM 编造病例库没有的检查结果；
* 若学生申请不存在的检查，系统可回答“本病例未提供该项检查结果”，并提示选择常规可用检查；
* 检查选择会进入评分。

## 4.5 OSCE 评分模块

### 目标

用结构化 rubric 对训练过程进行评分。

### 推荐总分结构

| 维度     | 分值 | 评分依据                |
| ------ | -: | ------------------- |
| 问诊完整性  | 25 | 关键病史项命中率            |
| 查体选择   | 15 | 是否请求关键查体            |
| 辅助检查选择 | 15 | 是否选择必要检查，是否过度检查     |
| 主要诊断   | 15 | 是否命中标准诊断或合理同义表达     |
| 鉴别诊断   | 15 | 是否覆盖关键鉴别诊断          |
| 推理表达   | 15 | 是否能把症状、体征、检查和诊断连接起来 |

### 评分策略

采用**规则评分 + LLM 语义评分**混合模式：

| 类型     |  占比 | 用途                  |
| ------ | --: | ------------------- |
| 规则评分   | 70% | 问诊项、查体项、检查项、诊断命中    |
| LLM 评分 | 30% | 推理表达、证据链完整性、鉴别诊断合理性 |

评分必须可追溯：每一项分数都要能对应到学生说过什么、系统披露了什么、rubric 中的哪一条。

### 4.5.0 Step 5 分阶段实现策略（2026-04-24 补充）

Step 4 已经跑通后端最小训练闭环，Step 5 不应一开始就把所有评分都交给 LLM。更稳妥的实现顺序是：

1. **第一小单元：规则评分与报告 API**
   - 实现 `evaluation_node` 的规则评分路径；
   - 先覆盖 `intent_keyword`、`exam_code`、`test_code`、`diagnosis_concept` 四类可审计规则；
   - 暴露 `/api/sessions/{session_id}/report`；
   - 输出 `rubric_scores`、`missed_items`、`total_score` 与简短结构化反馈；
   - 不调用 LLM，保证测试稳定、可复现。
2. **第二小单元：推理覆盖与 LLM 评分**
   - 接入 `reasoning_coverage` 与 `llm_rubric`；
   - LLM 只判断学生推理文本是否覆盖 rubric 要求的 evidence，不引入病例外医学事实；
   - LLM 输出必须符合 `LlmRubricResponse` 结构，便于追踪和回放。
3. **第三小单元：反馈生成**
   - 实现 `feedback_node`；
   - 把规则评分、LLM 评分、漏项和已披露证据转成面向学生的教学反馈；
   - 反馈只用于学习建议，不给真实诊疗建议。

这个分阶段策略的核心是：**结构化病例和 rubric 决定事实与评分边界，LLM 负责语义判断和表达增强。**

### 4.5.1 Rubric 顶层 Pydantic 模型（v1.1 新增）

```python
from __future__ import annotations
from typing import Literal
from pydantic import BaseModel, Field

DimensionId = Literal[
    "history_taking",          # 25 分
    "physical_exam",           # 15 分
    "auxiliary_test",          # 15 分
    "main_diagnosis",          # 15 分
    "differential_diagnosis",  # 15 分
    "reasoning",               # 15 分
]

ScoringMode = Literal["rule", "llm", "hybrid"]

class RubricItem(BaseModel):
    item_id: str = Field(..., pattern=r"^[a-z][a-z0-9_]*$")
    description: str                           # 人类可读, 反馈中会引用
    max_score: int = Field(..., ge=1, le=25)
    match_rule: "MatchRule"
    evidence_expected: list[str] = []          # fact_id / exam_code / test_code / point_id

class RubricDimension(BaseModel):
    dimension_id: DimensionId
    weight: int                                # 维度总分 (对应 §4.5 总分结构)
    scoring_mode: ScoringMode
    items: list[RubricItem]

class Rubric(BaseModel):
    rubric_id: str = Field(..., pattern=r"^[a-z0-9_]+_\d{3}_rubric(_[a-z]+)?$")
    case_id: str
    version: str = "v1"
    total_score: int = 100
    dimensions: list[RubricDimension]
    schema_version: Literal["1.1"] = "1.1"
```

**设计要点**

- `dimension_id` 用 `Literal` 严格枚举，对齐 §4.5 推荐总分结构的 6 个维度，防止新病例漏维度或拼错。
- `total_score` 固定 100：Step 2 校验器会断言 `sum(dim.weight for dim in dimensions) == 100`。
- `scoring_mode` 用于高层快速识别"这个维度该走规则、走 LLM、还是混合"：`history_taking / physical_exam / auxiliary_test / main_diagnosis / differential_diagnosis` 默认 `rule`，`reasoning` 默认 `llm`，`hybrid` 允许单维度内同时挂 `intent_keyword` 规则项和 `llm_rubric` 兜底项。

### 4.5.2 MatchRule：6 种机读匹配规则（v1.1 新增）

v1.0 §4.5 只说"规则评分 70% / LLM 评分 30%"。v1.1 把"规则评分"具体化到**可执行的 6 种 matcher**，每种 `kind` 有明确的 `spec` 形状与评分算法，消除"规则评分到底是什么规则"的歧义。

```python
MatchKind = Literal[
    "intent_keyword",        # 学生问过包含特定 topic+slot 的问题
    "exam_code",             # 学生请求过此 exam_code
    "test_code",             # 学生请求过此 test_code
    "diagnosis_concept",     # 学生提交的主/鉴别诊断命中 (含同义词)
    "reasoning_coverage",    # 学生推理表达覆盖 required_evidence
    "llm_rubric",            # 交给 LLM 评, 走 §4.5.3 契约
]

class MatchRule(BaseModel):
    kind: MatchKind
    spec: dict                                 # 类型相关参数, 下表定义
```

| kind                  | `spec` 必填字段                                                                                 | 评分算法                                                                                 |
| --------------------- | --------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------- |
| `intent_keyword`      | `topic: HistoryTopic`、`slot: HistorySlot \| None`、`any_of_keywords: list[str]`                | 在 `state.asked_questions` 中模糊匹配任一关键词；任意命中 → `max_score`，否则 0           |
| `exam_code`           | `exam_code: str`、`must: bool`                                                                 | `state.requested_exams` 含该 code → `max_score`；`must=True` 且漏 → 0                     |
| `test_code`           | `test_code: str`、`must: bool`、`deduct_if_forbidden: int = 0`                                  | 同上；若该 code 属 `forbidden_items` 被请求，则从该维度总分扣 `deduct_if_forbidden`         |
| `diagnosis_concept`   | `target: str`、`synonyms: list[str]`、`icd10_hint: str \| None`                                 | 对 `state.final_submission.main_diagnosis` 做规范化后精确或模糊匹配 (synonyms)               |
| `reasoning_coverage`  | `required_evidence: list[str]`、`min_coverage_ratio: float = 0.6`                              | 覆盖比例 `covered / required`；≥ ratio → `max_score`；否则按覆盖比例线性打分                   |
| `llm_rubric`          | `prompt_id: str`、`max_score: int`                                                             | 走 §4.5.3 Prompt 契约，以 `LlmRubricResponse.score` 为最终分                              |

**评分优先级**：同一 `RubricItem` 仅绑定一个 `MatchRule`；若需要"规则命中先算，未命中交 LLM"，在 rubric 里拆成 2 个并列 Item，或使用 `llm_rubric` 的 prompt 显式让 LLM 读规则结果并判定。不引入嵌套规则，保持 matcher 原子性。

**规则 / LLM 占比控制**：v1.0 §4.5 的 70% / 30% 由 `sum(max_score) of rule items / total_score == 0.7` 在 rubric 静态校验阶段（§8.4.1）自动断言，不依赖运行时平均。

### 4.5.3 LLM 语义评分的 Prompt 契约（v1.1 新增）

```python
class LlmRubricRequest(BaseModel):
    rubric_item_id: str
    description: str                           # RubricItem.description
    max_score: int
    student_final_reasoning: str               # 学生提交的推理表达原文
    relevant_facts_revealed: list[str]         # 本次 session 已披露的 fact_id
    required_evidence: list[str]               # RubricItem.evidence_expected

class LlmRubricResponse(BaseModel):
    score: int = Field(..., ge=0)
    covered_evidence: list[str]                # 子集 ⊆ required_evidence
    missing_evidence: list[str]                # required_evidence 的其余部分
    rationale: str = Field(..., max_length=120)
```

**系统提示（模板）**

```text
你是 OSCE 临床思维训练的评分员。你只能依据输入中列出的 required_evidence 和
学生的 student_final_reasoning 打分。你不得引入输入之外的医学事实。

评分规则：
- 满分为 max_score。
- 学生推理表达每覆盖一项 required_evidence 得 (max_score / len(required_evidence)) 分，
  四舍五入到整数; score 不得超过 max_score。
- 若学生推理包含不在 relevant_facts_revealed 中的"事实编造"，rationale 必须
  指出，并不得因此加分。
- rationale 不超过 120 字, 不输出具体用药方案、剂量或真实诊疗建议。
```

**用户提示（模板）**：直接把 `LlmRubricRequest` 序列化为 JSON 给模型，要求以 `LlmRubricResponse` 的 schema 返回。

**调用方式（LangGraph 节点内）**

```python
from langchain_anthropic import ChatAnthropic
from app.models.rubric import LlmRubricRequest, LlmRubricResponse

_llm = ChatAnthropic(model="claude-sonnet-4-6").with_structured_output(LlmRubricResponse)

def llm_rubric_score(req: LlmRubricRequest) -> LlmRubricResponse:
    return _llm.invoke([
        {"role": "system", "content": SYSTEM_PROMPT_TEMPLATE},
        {"role": "user", "content": req.model_dump_json()},
    ])
```

使用 `with_structured_output` 保证返回必为合法 `LlmRubricResponse`，避免 JSON 解析失败。

## 4.6 反馈报告模块

### 目标

### 报告结构

## 4.7 受控自我进化模块

### 目标

吸收 GA 的优秀特性，但规避其安全风险。

本项目的“自我进化”不是让系统自己改医学知识，而是让系统从训练日志中发现教学模式，并沉淀为可审核的教学 Skill。

### 进化对象

允许进化：

* 提示策略；
* 错误模式识别规则；
* 训练路径推荐；
* 反馈表达模板；
* 某类病例的教学纠偏 SOP；
* 学生画像标签。

不允许进化：

* 标准诊断；
* 病例事实；
* 检查结果；
* 医学知识结论；
* 用药建议；
* 医疗处置建议。

### Skill 生命周期

```text
训练日志
→ 错误模式聚类
→ 生成候选 Skill
→ 自动回归测试
→ 管理端审核
→ 启用 Skill
→ 后续病例训练调用
→ 继续评估效果
```

### Skill 字段

| 字段                  | 说明                                           |
| ------------------- | -------------------------------------------- |
| `skill_id`          | Skill 唯一 ID                                  |
| `trigger_condition` | 触发条件，如“腹痛病例中漏问泌尿系统症状”                        |
| `teaching_action`   | 教学动作，如“给系统性追问提示”                             |
| `applicable_cases`  | 适用病例标签                                       |
| `evidence_sessions` | 触发该 Skill 的训练记录                              |
| `expected_effect`   | 预期改善指标                                       |
| `test_result`       | 回归测试结果                                       |
| `status`            | candidate / approved / rejected / deprecated |
| `review_note`       | 审核说明                                         |

5. 所有工具调用必须写入日志；
6. 禁止线上开放任意代码执行工具。

## 8.3 病例生成原则

### 允许

* 从公开 OSCE 对话中学习问诊流程；
* 从公开病例推理数据中抽取诊断依据；
* 将公开英文病例改写成中文教学病例；
* 自行补充查体和检查结果，但必须与诊断逻辑一致；
* 标明“基于公开数据改写，不对应真实患者”。

### 禁止

* 直接复制版权不明网页病例作为产品数据；
* 使用真实患者个人信息；
* 使用未经许可的教材全文；
* 把复杂罕见病例直接给初学者训练；
* 让 LLM 自动生成病例后不做校验就入库；
* 不标来源地展示公开数据加工结果。

## 8.4 病例质量校验

每个病例上线前必须通过：

| 校验项     | 标准                        |
| ------- | ------------------------- |
| 字段完整性   | 主诉、病史、查体、检查、诊断、rubric 均存在 |
| 诊断一致性   | 病史、体征、检查能支持主要诊断           |
| 鉴别诊断合理性 | 至少 2 个合理鉴别                |
| 信息披露控制  | 患者回答不提前泄露诊断               |
| 检查结果一致性 | 检查结果不能与病例诊断矛盾             |
| 评分可执行   | rubric 能映射到学生行为           |
| 来源完整性   | 每个病例有 source attribution  |
| 安全声明    | 标明仅用于教学模拟                 |

### 8.4.1 校验器规则 → Pydantic validator 实现点（v1.1 新增）

把上表 8 条校验项一一对应到 `app/validators/case_validator.py` 的具体实现点。Step 2 编写校验器时逐条落地；病例必须双重通过"Pydantic 结构校验 + 业务 validator"才能进 `data/cases/`。

| 校验项         | 实现方式                                                                                                                                     |
| ----------- | ---------------------------------------------------------------------------------------------------------------------------------------- |
| 字段完整性       | 依赖 Pydantic 必填约束 + `Case.model_validate(json_data)`；缺字段即抛 `ValidationError`                                                             |
| 诊断一致性       | `@model_validator(mode="after")` on `Case`：`len([p for p in diagnosis.reasoning_points if p.kind == "支持"]) >= 3`                          |
| 鉴别诊断合理性     | `DifferentialDiagnosis` 的 `min_length=2` 由 `DiagnosisSpec.differential_diagnoses: list[...] = Field(..., min_length=2)` 保证               |
| 信息披露控制      | `@model_validator(mode="after")` on `Case`：`hidden_facts[].canonical_answer` 不含 `main_diagnosis` 或 `main_diagnosis_synonyms` 任一字面值       |
| 检查结果一致性     | `@model_validator(mode="after")` on `Case`：每个 `is_abnormal=True` 的 `exam_code / test_code` 至少出现在一个 `reasoning_points[].required_evidence` |
| 评分可执行       | `@model_validator(mode="after")` on `Rubric` + `Case`：每条 `RubricItem.evidence_expected` 中的 id 都必须能在 case 的 facts / exams / tests / reasoning_points 中找到 |
| 来源完整性       | `SourceAttribution.source_id` 必须在 `data/attribution/source_registry/sources.json` 中存在（运行时加载一次做集合校验）                                       |
| 安全声明        | `safety_notes: str = Field(..., min_length=1)`；额外在 validator 中检查不得出现"可替代医生诊断"等禁用短语                                                        |

**补充校验（v1.1 Rubric 特有）**

| 校验项                 | 实现方式                                                                                                                       |
| ------------------- | -------------------------------------------------------------------------------------------------------------------------- |
| Rubric 维度总分         | `@model_validator(mode="after")` on `Rubric`：`sum(d.weight for d in dimensions) == total_score`                            |
| 维度单项分上限             | 每条 `RubricItem.max_score <= dimension.weight`                                                                              |
| 规则评分占比（v1.0 §4.5 的 70%） | `sum(item.max_score for item in 所有非 llm_rubric Item) / total_score >= 0.65` （下限 65%，给微调留余地）                       |
| MatchRule.spec 形状   | `@field_validator("spec")` on `MatchRule`：按 `kind` 检查 `spec` 必填键是否齐全（缺 `exam_code` 时 `kind="exam_code"` 直接拒绝）         |
| Evidence 存在性        | rubric 的 `evidence_expected` ⊆ case 的 ids 集合 (`case + rubric` 绑定校验，Step 2 的 `validate_case_rubric_pair` 函数承载)          |

# 9. RAG 设计

## 9.1 RAG 的定位

RAG 不作为项目主角。主角是 OSCE 状态训练，RAG 只服务于：

1. 反馈解释；
2. 知识点推荐；
3. 评分依据引用；
4. 病例相似检索；
5. 答辩时证明可追溯。

## 9.2 向量库划分

| 向量库               | 内容               | 用途         |
| ----------------- | ---------------- | ---------- |
| `case_store`      | 病例摘要、症状、诊断、鉴别诊断  | 相似病例推荐     |
| `rubric_store`    | 评分条目和解释          | 评分解释       |
| `knowledge_store` | 问诊框架、症状学知识、诊断学基础 | 反馈补充       |
| `skill_store`     | 已审核教学 Skill      | 个性化提示和训练推荐 |

## 9.3 检索策略

| 场景     | 检索内容                              |
| ------ | --------------------------------- |
| 学生请求提示 | 当前病例 + 已收集线索 + 相关教学 Skill         |
| 学生提交诊断 | 当前病例 rubric + 标准 reasoning points |
| 生成反馈报告 | 漏项 + 相关知识点 + 下一病例                 |
| 推荐下一病例 | 学生薄弱项 + 病例标签                      |
| 教学进化   | 多次日志中的错误模式 + 已有 Skill             |

## 9.4 输出引用规范

反馈报告中的解释必须有“依据来源”，例如：

```text
依据类型：病例 rubric
依据 ID：abdominal_pain_001.rubric.history.03
说明：腹痛问诊应追问消化道和泌尿系统伴随症状。
```

这不是代码实现，而是产品输出格式要求。

# 14. 安全与合规设计

## 14.1 安全边界

系统必须在首页、训练页、报告页展示：

> 本系统仅用于医学教育模拟训练，不用于真实疾病诊断、治疗决策、用药指导或急救处置。若存在真实健康问题，请咨询合格医疗专业人员或及时就医。

## 14.2 拒答场景

| 场景           | 系统行为                  |
| ------------ | --------------------- |
| 用户询问真实症状该怎么办 | 拒绝给诊疗结论，建议就医          |
| 用户要求用药剂量     | 拒绝提供具体用药方案            |
| 用户上传真实患者隐私   | 提示不要输入个人敏感信息          |
| 用户要求系统替代医生诊断 | 强调教学模拟，不做真实诊断         |
| 用户询问急症处理     | 建议立即联系当地急救服务          |
| 用户要求直接给病例答案  | 在训练阶段拒绝透露，提示继续问诊或提交诊断 |

## 14.3 数据安全

* 不采集真实姓名、身份证、手机号；
* 学生 ID 可匿名；
* 日志中不保存真实患者信息；
* 公开数据保留来源和许可；
* 管理端限制访问；
* 所有 LLM 调用记录需要脱敏；
* 训练报告可导出，但不包含敏感身份信息。

# 15. 系统评测方案

## 15.1 技术评测

| 指标         | 目标         |
| ---------- | ---------- |
| 信息提前泄露率    | < 5%       |
| 问诊意图识别准确率  | ≥ 85%      |
| 查体/检查返回一致性 | ≥ 95%      |
| 同一输入评分稳定性  | ≥ 95%      |
| 安全越界拒答率    | ≥ 95%      |
| 反馈引用覆盖率    | ≥ 90%      |
| 单轮响应延迟     | 主要交互 < 5 秒 |
| 报告生成时间     | < 30 秒     |

## 15.2 教学评测

| 指标        | 说明               |
| --------- | ---------------- |
| 病例完成率     | 用户是否能完整完成一次训练    |
| 平均训练时长    | 判断 OSCE 训练流程是否合理 |
| 问诊覆盖率     | 学生问到关键项目比例       |
| 鉴别诊断覆盖率   | 是否提出合理鉴别         |
| 推理链完整度    | 是否能解释诊断依据        |
| 反馈可理解度    | 用户主观评分           |
| 下一病例推荐命中度 | 推荐是否对应该学生薄弱点     |

## 15.3 回归测试集

至少构造三类标准路径：

| 路径     | 说明             | 预期  |
| ------ | -------------- | --- |
| 优秀学生路径 | 问诊完整、检查合理、诊断正确 | 高分  |
| 普通学生路径 | 主要诊断正确但漏问部分病史  | 中高分 |
| 错误学生路径 | 问诊不足、检查混乱、诊断错误 | 低分  |

每次修改 prompt、病例、rubric 或 Skill 后都要跑回归测试。

# 16. 演示设计

## 16.1 首选 Demo 病例

推荐病例：**右下腹痛 / 急性阑尾炎教学病例**

原因：

* 问诊链条清楚；
* 查体和辅助检查容易展示；
* 鉴别诊断明确；
* 学生容易漏问；
* 评分点好设计；
* 观众容易理解。

## 16.2 Demo 展示顺序

1. 打开首页，说明系统定位；
2. 进入病例选择页；
3. 选择“腹痛模块：右下腹痛病例”；
4. 展示主诉；
5. 学生开始问诊；
6. 虚拟患者按病例回答；
7. 学生申请查体；
8. 系统返回腹部压痛、反跳痛等结果；
9. 学生申请血常规、腹部超声；
10. 系统返回结构化检查结果；
11. 学生提交主要诊断和鉴别诊断；
12. 系统生成 OSCE 评分报告；
13. 展示漏项和个性化建议；
14. 切到管理端看训练日志；
15. 展示系统从多次训练中挖掘出“腹痛问诊遗漏泌尿系统症状”的候选 Skill；
16. 审核并启用 Skill；
17. 回到新训练，展示 Skill 对提示策略的影响。

## 16.3 Demo 必须避免

* 现场临时自由发挥太多；
* 让系统回答真实医疗问题；
* 展示复杂罕见病导致评委看不懂；
* 只展示聊天，不展示评分和日志；
* 把自我进化说成“系统自己学医学”。

# 19. 风险与处理方案

| 风险           | 严重性 | 处理                             |
| ------------ | --: | ------------------------------ |
| 没有老师导致医学审核不足 |   高 | 明确标注公开数据原型；正式参赛前尽量找教师/医师挂名主讲   |
| 数据来源版权不清     |   高 | 只使用许可明确来源；免费网页只参考不入库           |
| 系统像普通问答      |  极高 | 必须突出状态机、信息披露、工具调用、评分报告         |
| LLM 编造病例事实   |   高 | 病例事实只从结构化数据取，LLM 只负责表达         |
| 自我进化被质疑危险    |   高 | 只进化教学策略，不进化医学事实                |
| 评分不稳定        |  中高 | 规则评分占主，LLM 只评语义和推理             |
| RAG 幻觉       |  中高 | RAG 不决定诊断，只做解释和引用              |
| Demo 不稳定     |   高 | 准备固定病例、固定输入、固定回放               |
| 项目过大失控       |   中 | 主线只围绕 OSCE 问诊训练，不扩展到导诊、用药、真实临床 |
| 公开数据英文为主     |   中 | 中文改写后保留原始来源；重点做教学病例而非数据复刻      |

## 24.5.1 为什么 raw 数据不能一次性“全理好”

当前结论是：**需要清洗，但不适合在 MVP 阶段一次性把 `data/raw/` 全量整理成可运行病例库。**

原因如下：

1. `data/raw/` 是公开来源的原料层，不是本项目运行时直接读取的病例层；运行时真正使用的是 `data/cases/*.json` 与 `data/rubrics/*.yaml`。
2. 不同来源的结构差异很大：有的是对话转录，有的是推理语句，有的是病例报告，有的甚至只是参考工程字段，必须先做抽取、对齐、翻译、裁剪与 schema 映射。
3. 本项目要求的是“教学模拟病例”，不是“公开数据逐字搬运”；因此必须做中文改写、教学合理性筛选、安全边界补充、rubric 配对，而不是直接入库。
4. 并非所有已下载数据都适合 MVP。比如 `case_report_collective_2025` 在来源登记中已明确为“仅作为二期扩展数据源，MVP 不直接入库”。
5. 一次性全量整理成本高、噪声大，而且会把大量暂时不用、质量不稳或不适合基础训练的病例提前带入系统，反而拖慢 Step 3 与 Step 4 的验证节奏。

因此更合理的策略是：**先做“小规模高质量”**，用 5 个左右常见教学病例把 schema、校验、读取流程和训练闭环跑通；之后再逐步扩充半自动清洗脚本和病例规模。
