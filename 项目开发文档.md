# 《问诊推理舱》项目开发文档 v1.0

项目全名：**问诊推理舱：基于公开 OSCE 病例数据的诊断学临床思维训练智能体**
推荐赛道：**课程智能体**
主技术基座：**LangGraph（LG）**
前端基座：**Agent Chat UI 改造为 OSCE 工作台**
后端基座：**agent-service-toolkit 改造为 LangGraph + FastAPI 服务**
医学虚拟标准化病人参考：**EasyMED**
GA 采用方式：**不作为主框架，只吸收“分层记忆、Skill 沉淀、自我反思进化”思想**

---

## 0. 文档边界与核心判断

本文档按“可开发、可演示、可答辩、可扩展”的完整项目规格编写，不按个人开发者身份压缩项目完整性。项目目标不是做一个医学问答机器人，而是做一个**诊断学课程训练智能体**：学生通过多轮问诊、查体申请、辅助检查申请、诊断提交和复盘反馈完成一次 OSCE 式临床思维训练。

本项目的数据策略默认不依赖校内教师或私有医院数据，而是使用**公开数据、公开病例、公开开源项目和自行结构化加工病例**。但必须说明：没有医学教师参与时，系统只能定位为**教学原型 / 比赛作品 / 模拟训练工具**，不能声称已完成正式课程医学审核。

赛事公开通知强调“人工智能+医学教学”融合创新、医学教育数字化转型和智能化教学能力提升，这决定项目必须体现教学载体、教学环节、实施路径和可展示效果，而不是单纯展示 Agent 技术。赛事通知地址：`https://www.cfxy.cn/yxb/tzgg/81c3510906174816a858d450261d12fb.htm`。 ([cfxy][1])

# 1. 项目概述

## 1.1 项目定位

**问诊推理舱**是一个面向医学生的诊断学临床思维训练智能体。系统通过虚拟标准化病人模拟、病例状态控制、问诊意图识别、查体/检查工具调用、OSCE rubric 评分和个性化反馈，帮助学生练习“如何一步步问出关键病史并形成诊断推理”。

项目不解决真实患者诊疗问题，不提供真实医疗建议，不替代医生和教师。所有病例均为公开数据改写或虚构教学病例，系统输出仅用于医学教育训练。

## 1.2 一句话价值主张

**让医学生不只是“知道疾病答案”，而是在智能体陪练中完成一次可记录、可评分、可复盘、可迭代的诊断学问诊训练。**

## 1.3 推荐赛道

**课程智能体。**

不要报“临床技能训练智能体”。原因是临床技能训练赛道天然要求高仿真临床场景，容易被评委追问医学准确性、临床责任、操作技能真实性。课程智能体更适合本项目，因为它可以绑定“诊断学 / 临床思维训练 / 内科学见习前训练”课程，重点展示教学目标、课程内容、评价方式和学生学习闭环。

## 1.4 核心用户

| 用户          | 角色定位      | 主要需求                           |
| ----------- | --------- | ------------------------------ |
| 医学生         | 训练者       | 练习问诊、查体选择、辅助检查选择、鉴别诊断和诊断表达     |
| 项目开发者 / 管理者 | 系统维护者     | 管理公开病例、配置 rubric、查看训练日志、评估系统表现 |
| 未来课程教师      | 审核者 / 使用者 | 审核病例、调整评分表、查看班级共性问题            |
| 评委          | 体验者       | 判断项目是否真实解决医学教学问题、是否有智能体特征和可推广性 |

地址：`https://docs.langchain.com/oss/python/langgraph/overview`

## 2.2 公开医学数据与训练资料

### 2.2.1 Fareez OSCE 医患访谈数据集

论文地址：`https://www.nature.com/articles/s41597-022-01423-1`
数据地址：`https://doi.org/10.6084/m9.figshare.c.5545842.v1`

用途：用于学习 OSCE 式医患问诊话轮、语言风格、问诊流程和症状采集结构。该数据集包含 OSCE 格式的模拟医患对话，数据记录包括 272 个 mp3 音频文件和 272 个对应转录文本文件，并覆盖呼吸、胃肠、心血管、肌骨、皮肤等类别。([Nature][8])

许可与限制：论文为 CC BY 4.0 开放许可，可在适当署名和标明修改的前提下使用、分享和改编；但数据主要是问诊对话，不包含完整查体、辅助检查和治疗闭环。([Nature][8])

本项目采用方式：

* 用作问诊语言和对话结构参考；
* 用作意图识别和症状抽取样例；
* 不直接把原始对话当作完整训练病例；
* 需要补充结构化病例、查体、检查和 rubric。

### 2.2.3 MedCaseReasoning

论文地址：`https://arxiv.org/abs/2505.11733`
GitHub 地址：`https://github.com/kevinwu23/Stanford-MedCaseReasoning`
HuggingFace 地址：`https://huggingface.co/datasets/zou-lab/MedCaseReasoning`

* 不建议直接把复杂病例报告作为新手 OSCE 训练病例，因为病例报告常偏罕见、复杂、不典型。

用途：二期扩展病例来源。CaseReportCollective 是从开放获取病例报告中抽取的结构化医学病例数据集，规模为 85,961 个 case reports，跨 14 个医学领域。([ACL Anthology][11])

# 3. 总体产品设计

## 3.1 核心训练流程

学生进入系统后，完成如下流程：

```text
选择课程模块
→ 进入病例
→ 阅读主诉
→ 多轮问诊
→ 请求查体
→ 请求辅助检查
→ 提交主要诊断
→ 提交鉴别诊断
→ 提交诊断依据
→ 系统评分
→ 系统反馈
→ 生成学习报告
→ 进入下一病例推荐
```

系统在学生提交诊断前不能直接泄露标准答案。系统只能根据学生问到的内容逐步开放病例信息。

## 3.2 产品核心差异

| 普通医学问答系统    | 问诊推理舱                   |
| ----------- | ----------------------- |
| 回答医学知识问题    | 模拟完整问诊训练                |
| 学生问什么，系统答什么 | 系统维护病例状态和训练阶段           |
| 不记录过程质量     | 记录问诊、查体、检查和诊断推理路径       |
| 主要评价最终答案    | 评价全过程：问诊、检查选择、鉴别诊断、推理表达 |
| 知识库是核心      | OSCE 训练状态机是核心，RAG 只是辅助  |
| 无进化机制       | 从训练日志中沉淀教学 Skill        |

## 3.3 项目边界

### 必须做

| 功能       | 说明                             |
| -------- | ------------------------------ |
| 虚拟标准化病人  | 能根据结构化病例多轮回答学生问题               |
| 问诊意图识别   | 判断学生是在问病史、请求查体、请求检查、请求提示还是提交诊断 |
| 信息披露控制   | 只回答学生问到的信息，不提前泄露诊断             |
| 查体/检查工具  | 学生请求后返回结构化查体或辅助检查结果            |
| OSCE 评分  | 按病例 rubric 对问诊、检查、诊断、推理评分      |
| 个性化反馈    | 输出优势、漏项、错误推理、下一步训练             |
| 公开数据来源追踪 | 每个病例和知识点都要有来源记录                |
| 训练日志     | 保存学生路径、系统回复、工具调用和评分结果          |
| 受控进化     | 从日志中生成教学 Skill 候选，经审核后启用       |

### 不做

| 不做项           | 原因                    |
| ------------- | --------------------- |
| 真实患者导诊        | 医疗风险高，偏离课程智能体         |
| 用药建议          | 合规风险高                 |
| 接入医院 HIS/EMR  | 个人开发无法获得稳定授权          |
| 真实临床决策支持      | 责任边界过重                |
| 影像识别/手术模拟     | 工程成本高，且不利于稳定演示        |
| 任意代码执行型 Agent | 医学教学系统不需要，安全风险过高      |
| 自动修改医学事实      | 自我进化只能改教学策略，不能改病例标准答案 |

# 4. 功能需求规格

## 4.1 病例库模块

### 目标

管理公开来源病例，将其转成可训练的 OSCE 病例卡。

### 功能

| 功能     | 描述                          | 优先级 |
| ------ | --------------------------- | --- |
| 病例列表   | 显示病例标题、系统模块、难度、标签、来源        | P0  |
| 病例详情   | 展示公开信息、隐藏信息、查体、检查、诊断、rubric | P0  |
| 病例导入   | 支持 JSON/CSV 导入公开加工病例        | P0  |
| 病例校验   | 检查字段完整性、诊断一致性、来源完整性         | P0  |
| 病例版本管理 | 同一病例可有 v1/v2，用于回归比较         | P1  |
| 病例来源追踪 | 保存原始公开来源、许可、加工说明            | P0  |

### 病例字段设计

| 字段                       | 说明             |
| ------------------------ | -------------- |
| `case_id`                | 病例唯一 ID        |
| `case_title`             | 病例标题           |
| `course_module`          | 课程模块，如腹痛、发热、胸痛 |
| `difficulty`             | 初级 / 中级 / 高级   |
| `chief_complaint`        | 主诉，训练开始时开放     |
| `patient_profile`        | 年龄、性别、职业等      |
| `hidden_history`         | 隐藏病史，学生问到才开放   |
| `physical_exam`          | 查体结果           |
| `auxiliary_tests`        | 血常规、影像、尿检等辅助检查 |
| `main_diagnosis`         | 标准主要诊断         |
| `differential_diagnoses` | 标准鉴别诊断         |
| `reasoning_points`       | 关键诊断依据         |
| `rubric`                 | 评分表            |
| `safety_notes`           | 教学模拟声明和真实诊疗限制  |
| `source_attribution`     | 来源地址、许可、改写说明   |

---

## 4.2 OSCE 训练模块

### 目标

让学生像面对标准化病人一样进行问诊，而不是直接索取答案。

### 功能

| 功能     | 描述                  | 优先级 |
| ------ | ------------------- | --- |
| 进入病例   | 展示主诉和初始任务           | P0  |
| 多轮问诊   | 学生自由输入问题，虚拟患者回答     | P0  |
| 当前阶段显示 | 显示问诊、查体、检查、诊断、反馈等阶段 | P0  |
| 已收集线索  | 自动列出学生已获得的关键信息      | P0  |
| 诊断假设草稿 | 学生可记录自己的初步假设        | P1  |
| 苏格拉底提示 | 学生卡住时给过程提示，不直接给答案   | P1  |
| 训练计时   | 模拟 OSCE 限时环境        | P2  |

### 回答原则

虚拟患者必须遵守：

1. 只扮演患者，不扮演医生；
2. 只回答病例中存在的信息；
3. 不主动说出未被问到的关键事实；
4. 不在诊断前透露标准诊断；
5. 对非本病例问题保持患者视角；
6. 对真实医疗问题触发安全边界；
7. 对学生要求“直接告诉我答案”时，只能提示继续问诊或提交诊断。

---

## 4.3 查体与辅助检查模块

### 目标

把“学生问一句，系统答一句”升级为“学生主动选择检查，系统按病例返回结果”。

### 功能

| 功能       | 描述              | 优先级 |
| -------- | --------------- | --- |
| 查体申请     | 学生选择或输入查体项目     | P0  |
| 检查申请     | 学生选择血常规、尿常规、影像等 | P0  |
| 结果返回     | 只返回病例库中定义的结果    | P0  |
| 不合理检查提示  | 过度或无关检查可提示其教学意义 | P1  |
| 成本/侵入性标签 | 标记检查是否基础、昂贵、有创  | P2  |

### 检查返回原则

* 结构化字段优先，LLM 只负责表达；
* 不允许 LLM 编造病例库没有的检查结果；
* 若学生申请不存在的检查，系统可回答“本病例未提供该项检查结果”，并提示选择常规可用检查；
* 检查选择会进入评分。

---

## 4.4 诊断提交模块

### 目标

要求学生在信息收集后主动形成诊断，而不是被系统直接喂答案。

### 提交内容

| 字段      | 说明                       |
| ------- | ------------------------ |
| 主要诊断    | 学生认为最可能的诊断               |
| 鉴别诊断    | 至少 2 个合理鉴别               |
| 支持依据    | 哪些病史、查体、检查支持主要诊断         |
| 排除依据    | 如何排除其他可能                 |
| 进一步处理建议 | 教学模拟下的下一步检查或处理方向，不涉及真实用药 |

---

## 4.5 OSCE 评分模块

### 目标

用结构化 rubric 对训练过程进行评分。

### 推荐总分结构

| 维度     | 分值 | 评分依据                |
| ------ | -: | ------------------- |
| 问诊完整性  | 25 | 关键病史项命中率            |
| 查体选择   | 15 | 是否请求关键查体            |
| 辅助检查选择 | 15 | 是否选择必要检查，是否过度检查     |
| 主要诊断   | 15 | 是否命中标准诊断或合理同义表达     |
| 鉴别诊断   | 15 | 是否覆盖关键鉴别诊断          |
| 推理表达   | 15 | 是否能把症状、体征、检查和诊断连接起来 |

### 评分策略

采用**规则评分 + LLM 语义评分**混合模式：

| 类型     |  占比 | 用途                  |
| ------ | --: | ------------------- |
| 规则评分   | 70% | 问诊项、查体项、检查项、诊断命中    |
| LLM 评分 | 30% | 推理表达、证据链完整性、鉴别诊断合理性 |

评分必须可追溯：每一项分数都要能对应到学生说过什么、系统披露了什么、rubric 中的哪一条。

---

## 4.6 反馈报告模块

### 目标

让学生知道自己不是“错了”，而是知道**错在哪里、为什么错、下次怎么练**。

### 报告结构

| 模块   | 内容                  |
| ---- | ------------------- |
| 总分   | 0—100               |
| 分项评分 | 问诊、查体、检查、诊断、鉴别、推理   |
| 优势   | 学生做得好的地方            |
| 漏项   | 未问到、未检查、未解释的关键点     |
| 推理问题 | 证据链断裂、过早闭合、鉴别不足等    |
| 补救建议 | 下一步学习点              |
| 推荐病例 | 根据错误模式推荐下一例         |
| 引用依据 | 对应病例来源、rubric、知识点来源 |

## 4.7 受控自我进化模块

### 目标

吸收 GA 的优秀特性，但规避其安全风险。

本项目的“自我进化”不是让系统自己改医学知识，而是让系统从训练日志中发现教学模式，并沉淀为可审核的教学 Skill。

### 进化对象

允许进化：

* 提示策略；
* 错误模式识别规则；
* 训练路径推荐；
* 反馈表达模板；
* 某类病例的教学纠偏 SOP；
* 学生画像标签。

不允许进化：

* 标准诊断；
* 病例事实；
* 检查结果；
* 医学知识结论；
* 用药建议；
* 医疗处置建议。

### Skill 生命周期

```text
训练日志
→ 错误模式聚类
→ 生成候选 Skill
→ 自动回归测试
→ 管理端审核
→ 启用 Skill
→ 后续病例训练调用
→ 继续评估效果
```

### Skill 字段

| 字段                  | 说明                                           |
| ------------------- | -------------------------------------------- |
| `skill_id`          | Skill 唯一 ID                                  |
| `trigger_condition` | 触发条件，如“腹痛病例中漏问泌尿系统症状”                        |
| `teaching_action`   | 教学动作，如“给系统性追问提示”                             |
| `applicable_cases`  | 适用病例标签                                       |
| `evidence_sessions` | 触发该 Skill 的训练记录                              |
| `expected_effect`   | 预期改善指标                                       |
| `test_result`       | 回归测试结果                                       |
| `status`            | candidate / approved / rejected / deprecated |
| `review_note`       | 审核说明                                         |

| `student_id`           | 学生 ID，可匿名              |

| `messages`             | 医患对话历史                 |
| `asked_questions`      | 学生问过的问题                |
| `intent_history`       | 每轮输入识别出的意图             |
| `revealed_facts`       | 已披露病例信息                |
| `requested_exams`      | 已请求查体                  |
| `requested_tests`      | 已请求辅助检查                |
| `student_hypotheses`   | 学生过程中的诊断假设             |

## 6.2 图节点设计

| 节点                        | 输入           | 输出       | 说明                 |
| ------------------------- | ------------ | -------- | ------------------ |
| `load_case_node`          | `case_id`    | 病例数据     | 加载病例公开信息和隐藏事实      |
| `input_router_node`       | 学生输入         | 意图类别     | 判断下一步走问诊、查体、检查、诊断等 |
| `intent_recognition_node` | 学生自然语言       | 结构化意图    | 识别问诊对象和槽位          |
| `patient_response_node`   | 意图 + 病例事实    | 患者回答     | 扮演虚拟患者，控制信息披露      |
| `physical_exam_node`      | 查体请求         | 查体结果     | 返回病例中已有查体结果        |
| `auxiliary_test_node`     | 检查请求         | 检查结果     | 返回病例中已有辅助检查        |
| `socratic_hint_node`      | 当前状态         | 提示       | 不泄露答案的过程提示         |
| `diagnosis_submit_node`   | 学生提交         | 诊断对象     | 固化最终诊断             |
| `evaluation_node`         | 全轨迹 + rubric | 分项分数     | 规则 + LLM 评分        |
| `feedback_node`           | 分数 + 漏项      | 反馈报告     | 生成学生报告             |
| `memory_update_node`      | 训练结果         | 学生画像     | 更新弱点标签             |
| `evolution_miner_node`    | 多次训练日志       | Skill 候选 | 挖掘教学改进建议           |
| `safety_guardrail_node`   | 越界输入         | 安全回复     | 拒绝真实医疗建议           |

## 6.3 阶段流转

```text
case_intro
→ history_taking
→ physical_exam
→ auxiliary_test
→ diagnosis_submission
→ evaluation
→ feedback
→ completed
```

允许跳转：

* 学生可在问诊阶段请求查体；
* 学生可在查体后继续问诊；
* 学生可在任意训练阶段请求“过程提示”；
* 学生提交诊断后，不允许继续补问关键病史，除非开启“复盘模式”。

## 7.1 工具清单

| 工具名                         | 类型     | 功能          | 是否必须 |
| --------------------------- | ------ | ----------- | ---- |
| `get_case_public_info`      | 数据工具   | 获取病例主诉和公开信息 | 必须   |
| `get_case_hidden_fact`      | 数据工具   | 按意图返回隐藏病史   | 必须   |
| `get_physical_exam_result`  | 数据工具   | 返回查体结果      | 必须   |
| `get_auxiliary_test_result` | 数据工具   | 返回辅助检查结果    | 必须   |
| `search_case_knowledge`     | RAG 工具 | 检索病例相关知识    | 必须   |
| `score_history_taking`      | 评分工具   | 评分问诊覆盖度     | 必须   |
| `score_test_selection`      | 评分工具   | 评分检查选择      | 必须   |
| `score_diagnosis_reasoning` | 评分工具   | 评分诊断和推理     | 必须   |
| `generate_feedback_report`  | 生成工具   | 生成反馈报告      | 必须   |
| `update_student_profile`    | 记忆工具   | 更新学生画像      | 建议   |
| `mine_error_patterns`       | 进化工具   | 挖掘共性错误      | 建议   |
| `propose_teaching_skill`    | 进化工具   | 生成候选 Skill  | 建议   |
| `approve_teaching_skill`    | 审核工具   | 启用 Skill    | 建议   |
| `safety_check`              | 安全工具   | 判断是否越界      | 必须   |

## 7.2 工具调用原则

1. 只让 LLM 决定“调用什么工具”，不让 LLM 编造工具结果；
2. 病例事实、查体结果、检查结果必须来自结构化病例库；
3. 评分必须输出依据；
4. 反馈必须引用病例 rubric 或知识库来源；
5. 所有工具调用必须写入日志；
6. 禁止线上开放任意代码执行工具。

# 8. 数据设计

## 8.1 数据来源登记表

每个外部数据源进入项目时必须登记：

| 字段                   | 说明                  |
| -------------------- | ------------------- |
| source_id            | 来源唯一 ID             |
| source_name          | 数据源名称               |
| source_url           | 地址                  |
| license              | 许可                  |
| data_type            | 对话 / 病例 / 推理 / 知识   |
| allowed_usage        | 可用于训练、评测、展示、参考中的哪一种 |
| transformation       | 是否翻译、改写、结构化         |
| attribution_required | 是否需要署名              |
| risk_note            | 风险说明                |

## 8.2 数据加工流程

```text
公开数据登记
→ 下载或手动整理
→ 来源许可检查
→ 去身份化检查
→ 结构化抽取
→ 中文转写/改写
→ 病例字段补全
→ rubric 生成
→ 医学一致性检查
→ 信息披露测试
→ 导入病例库
→ 构建向量索引
```

## 8.3 病例生成原则

### 允许

* 从公开 OSCE 对话中学习问诊流程；
* 从公开病例推理数据中抽取诊断依据；
* 将公开英文病例改写成中文教学病例；
* 自行补充查体和检查结果，但必须与诊断逻辑一致；
* 标明“基于公开数据改写，不对应真实患者”。

### 禁止

* 直接复制版权不明网页病例作为产品数据；
* 使用真实患者个人信息；
* 使用未经许可的教材全文；
* 把复杂罕见病例直接给初学者训练；
* 让 LLM 自动生成病例后不做校验就入库；
* 不标来源地展示公开数据加工结果。

## 8.4 病例质量校验

每个病例上线前必须通过：

| 校验项     | 标准                        |
| ------- | ------------------------- |
| 字段完整性   | 主诉、病史、查体、检查、诊断、rubric 均存在 |
| 诊断一致性   | 病史、体征、检查能支持主要诊断           |
| 鉴别诊断合理性 | 至少 2 个合理鉴别                |
| 信息披露控制  | 患者回答不提前泄露诊断               |
| 检查结果一致性 | 检查结果不能与病例诊断矛盾             |
| 评分可执行   | rubric 能映射到学生行为           |
| 来源完整性   | 每个病例有 source attribution  |
| 安全声明    | 标明仅用于教学模拟                 |

# 9. RAG 设计

## 9.1 RAG 的定位

RAG 不作为项目主角。主角是 OSCE 状态训练，RAG 只服务于：

1. 反馈解释；
2. 知识点推荐；
3. 评分依据引用；
4. 病例相似检索；
5. 答辩时证明可追溯。

## 9.2 向量库划分

| 向量库               | 内容               | 用途         |
| ----------------- | ---------------- | ---------- |
| `case_store`      | 病例摘要、症状、诊断、鉴别诊断  | 相似病例推荐     |
| `rubric_store`    | 评分条目和解释          | 评分解释       |
| `knowledge_store` | 问诊框架、症状学知识、诊断学基础 | 反馈补充       |
| `skill_store`     | 已审核教学 Skill      | 个性化提示和训练推荐 |

## 9.3 检索策略

| 场景     | 检索内容                              |
| ------ | --------------------------------- |
| 学生请求提示 | 当前病例 + 已收集线索 + 相关教学 Skill         |
| 学生提交诊断 | 当前病例 rubric + 标准 reasoning points |
| 生成反馈报告 | 漏项 + 相关知识点 + 下一病例                 |
| 推荐下一病例 | 学生薄弱项 + 病例标签                      |
| 教学进化   | 多次日志中的错误模式 + 已有 Skill             |

## 9.4 输出引用规范

反馈报告中的解释必须有“依据来源”，例如：

```text
依据类型：病例 rubric
依据 ID：abdominal_pain_001.rubric.history.03
说明：腹痛问诊应追问消化道和泌尿系统伴随症状。
```

这不是代码实现，而是产品输出格式要求。

# 14. 安全与合规设计

## 14.1 安全边界

系统必须在首页、训练页、报告页展示：

> 本系统仅用于医学教育模拟训练，不用于真实疾病诊断、治疗决策、用药指导或急救处置。若存在真实健康问题，请咨询合格医疗专业人员或及时就医。

## 14.2 拒答场景

| 场景           | 系统行为                  |
| ------------ | --------------------- |
| 用户询问真实症状该怎么办 | 拒绝给诊疗结论，建议就医          |
| 用户要求用药剂量     | 拒绝提供具体用药方案            |
| 用户上传真实患者隐私   | 提示不要输入个人敏感信息          |
| 用户要求系统替代医生诊断 | 强调教学模拟，不做真实诊断         |
| 用户询问急症处理     | 建议立即联系当地急救服务          |
| 用户要求直接给病例答案  | 在训练阶段拒绝透露，提示继续问诊或提交诊断 |

## 14.3 数据安全

* 不采集真实姓名、身份证、手机号；
* 学生 ID 可匿名；
* 日志中不保存真实患者信息；
* 公开数据保留来源和许可；
* 管理端限制访问；
* 所有 LLM 调用记录需要脱敏；
* 训练报告可导出，但不包含敏感身份信息。

# 15. 系统评测方案

## 15.1 技术评测

| 指标         | 目标         |
| ---------- | ---------- |
| 信息提前泄露率    | < 5%       |
| 问诊意图识别准确率  | ≥ 85%      |
| 查体/检查返回一致性 | ≥ 95%      |
| 同一输入评分稳定性  | ≥ 95%      |
| 安全越界拒答率    | ≥ 95%      |
| 反馈引用覆盖率    | ≥ 90%      |
| 单轮响应延迟     | 主要交互 < 5 秒 |
| 报告生成时间     | < 30 秒     |

## 15.2 教学评测

| 指标        | 说明               |
| --------- | ---------------- |
| 病例完成率     | 用户是否能完整完成一次训练    |
| 平均训练时长    | 判断 OSCE 训练流程是否合理 |
| 问诊覆盖率     | 学生问到关键项目比例       |
| 鉴别诊断覆盖率   | 是否提出合理鉴别         |
| 推理链完整度    | 是否能解释诊断依据        |
| 反馈可理解度    | 用户主观评分           |
| 下一病例推荐命中度 | 推荐是否对应该学生薄弱点     |

## 15.3 回归测试集

至少构造三类标准路径：

| 路径     | 说明             | 预期  |
| ------ | -------------- | --- |
| 优秀学生路径 | 问诊完整、检查合理、诊断正确 | 高分  |
| 普通学生路径 | 主要诊断正确但漏问部分病史  | 中高分 |
| 错误学生路径 | 问诊不足、检查混乱、诊断错误 | 低分  |

每次修改 prompt、病例、rubric 或 Skill 后都要跑回归测试。

# 16. 演示设计

## 16.1 首选 Demo 病例

推荐病例：**右下腹痛 / 急性阑尾炎教学病例**

原因：

* 问诊链条清楚；
* 查体和辅助检查容易展示；
* 鉴别诊断明确；
* 学生容易漏问；
* 评分点好设计；
* 观众容易理解。

## 16.2 Demo 展示顺序

1. 打开首页，说明系统定位；
2. 进入病例选择页；
3. 选择“腹痛模块：右下腹痛病例”；
4. 展示主诉；
5. 学生开始问诊；
6. 虚拟患者按病例回答；
7. 学生申请查体；
8. 系统返回腹部压痛、反跳痛等结果；
9. 学生申请血常规、腹部超声；
10. 系统返回结构化检查结果；
11. 学生提交主要诊断和鉴别诊断；
12. 系统生成 OSCE 评分报告；
13. 展示漏项和个性化建议；
14. 切到管理端看训练日志；
15. 展示系统从多次训练中挖掘出“腹痛问诊遗漏泌尿系统症状”的候选 Skill；
16. 审核并启用 Skill；
17. 回到新训练，展示 Skill 对提示策略的影响。

## 16.3 Demo 必须避免

* 现场临时自由发挥太多；
* 让系统回答真实医疗问题；
* 展示复杂罕见病导致评委看不懂；
* 只展示聊天，不展示评分和日志；
* 把自我进化说成“系统自己学医学”。

# 19. 风险与处理方案

| 风险           | 严重性 | 处理                             |
| ------------ | --: | ------------------------------ |
| 没有老师导致医学审核不足 |   高 | 明确标注公开数据原型；正式参赛前尽量找教师/医师挂名主讲   |
| 数据来源版权不清     |   高 | 只使用许可明确来源；免费网页只参考不入库           |
| 系统像普通问答      |  极高 | 必须突出状态机、信息披露、工具调用、评分报告         |
| LLM 编造病例事实   |   高 | 病例事实只从结构化数据取，LLM 只负责表达         |
| 自我进化被质疑危险    |   高 | 只进化教学策略，不进化医学事实                |
| 评分不稳定        |  中高 | 规则评分占主，LLM 只评语义和推理             |
| RAG 幻觉       |  中高 | RAG 不决定诊断，只做解释和引用              |
| Demo 不稳定     |   高 | 准备固定病例、固定输入、固定回放               |
| 项目过大失控       |   中 | 主线只围绕 OSCE 问诊训练，不扩展到导诊、用药、真实临床 |
| 公开数据英文为主     |   中 | 中文改写后保留原始来源；重点做教学病例而非数据复刻      |

# 22. 最关键的产品原则

这个项目必须始终坚持四条原则：

1. **状态优先**：系统核心不是聊天，而是 OSCE 训练状态机。
2. **事实受控**：病例事实来自结构化数据，不让模型随意编。
3. **评价闭环**：每次训练必须产生评分、漏项、建议和记录。
4. **进化受控**：只进化教学策略，不进化医学事实。

只要这四条做到，项目就能和普通医学问答、普通 RAG 知识库、普通病例库明显拉开差距。
